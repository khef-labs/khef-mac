KHEF Q&A

Anticipated questions and reasoning

Why Build This?

"Why not just use Notion / Confluence / a wiki?"

Short: Wikis aren't AI-native. They can't be queried by Claude mid-conversation.

  • No MCP integration — AI can't read/write to Notion during a coding session
  • No structured memory types — everything is just "a page"
  • No knowledge graph — can't traverse relationships between decisions
  • Wikis document what, not why — decisions need rationale
  • Wikis decay — Khef is written during work, not after

"Why not just write better documentation?"

Short: Docs are read by humans, memories are read by AI and humans.

  • Docs require discipline to maintain — memories are created as you work
  • AI can't query your docs mid-conversation (no MCP tools for Confluence)
  • Docs don't capture decision rationale or rejected alternatives
  • Khef's structured types (decision, pattern, context) enforce useful formats

"Isn't this just another notes app?"

Short: Notes apps don't have MCP tools, knowledge graphs, or polymorphic status systems.

  • 50+ MCP tools let Claude Code and Codex CLI read/write memories
  • Knowledge graph with 11 relation types connects memories
  • Status workflows per memory type (decisions, todos, patterns)
  • Agent rules sync to CLAUDE.md / AGENTS.md automatically
Tech Stack

"Why PostgreSQL instead of MongoDB / a document DB?"

Short: Memories are relational. The knowledge graph needs joins. Postgres has everything.

  • Knowledge graph = relations = foreign keys + joins
  • Full-text search built-in (tsvector, GIN indexes)
  • pgvector for semantic search — no separate vector DB needed
  • ACID transactions for consistent state
  • Battle-tested, great tooling, everyone knows it

"Why pgvector vs a dedicated vector DB like Pinecone?"

Short: Simpler ops, co-located data, good enough for this scale.

  • One less service to run and maintain
  • Vector embeddings live next to the memories they reference
  • For thousands of memories, pgvector performance is fine
  • Can still use Chroma or Qdrant if you want — it's configurable

"Why Fastify over Express?"

Short: Faster, better TypeScript support, built-in validation.

  • ~2x faster than Express in benchmarks
  • First-class TypeScript support
  • JSON Schema validation built-in
  • Plugin architecture keeps code organized

"Why Preact over React?"

Short: Same API, 3KB instead of 40KB, fast enough for a dashboard.

  • API-compatible with React (same hooks, same patterns)
  • Much smaller bundle size
  • This is a developer tool, not a complex app — Preact is plenty

"Why self-hosted instead of SaaS?"

Short: Your code context is sensitive. Your data stays on your machine.

  • No data leaving your network — important for enterprise
  • Save and remember without fear of judgement — we're all more forgetful than we let on
  • Works offline — no dependency on external services
  • No vendor lock-in — you own the data and can export anytime
  • Customizable — add your own memory types, integrations
  • Could become SaaS later, but self-hosted is the right default
Design Decisions

"Why Markdown for content?"

Short: Markdown is the best language for AI/human collaboration.

  • Human-readable and writable — no special tooling needed
  • AI models understand and generate it natively
  • Universal export/import — Notion, Obsidian, GitHub, wikis all support it
  • Portable — your content isn't locked in a proprietary format
  • Supports code blocks, diagrams (mermaid), tables — everything devs need

"Why structured memory types instead of free-form notes?"

Short: Structure enables workflows. Decisions have different lifecycles than todos.

  • Each type has its own status workflow (proposed→accepted, open→done)
  • Agents can filter by type ("show me all accepted decisions")
  • UI can render type-specific views (kanban for todos, timeline for decisions)
  • Forces useful structure without being rigid — content is still markdown

"Why a knowledge graph instead of just tags?"

Short: Tags are flat. Relations capture how things connect.

  • "Pattern X implements Decision Y" is richer than both having tag "auth"
  • Can traverse: "show me everything this decision depends on"
  • Graph health analysis finds orphan memories
  • 11 relation types cover most knowledge relationships
  • It's optional — as useful as you make it, or don't use it at all
  • You still benefit from a rich store of searchable (and semantically searchable) knowledge

"Why MCP instead of just a REST API?"

Short: MCP is how Claude Code and Codex CLI integrate with tools.

  • MCP is the standard protocol for AI tool integration
  • Tools appear natively in Claude/Codex — no copy-paste
  • REST API exists too — MCP server calls it internally
  • Best of both: AI-native + human-accessible
Practical Concerns

"How does this scale?"

Short: PostgreSQL scales fine. This isn't big data — it's knowledge.

  • 10,000 memories ≈ 500,000 lines of searchable knowledge — years of documentation
  • Storage is trivial: 100K memories ≈ 100MB
  • pgvector handles up to ~1M vectors efficiently
  • Full-text search is fast with GIN indexes
  • The limit isn't technical — it's cognitive

"What happens if you lose your data?"

Short: It happened to me. I recovered from session transcripts.

  • Claude in auto mode destroyed my Docker volume — a month of memories gone
  • Thought all was lost, then realized: session transcripts had everything
  • Claude read the sessions and used MCP tools to recreate the memories
  • Lesson: sessions are a backup, not just history — and MCP makes recovery possible
  • Session embedding/search is now implemented — your conversations are searchable

"What about security?"

Short: Self-hosted, localhost by default, no external dependencies.

  • Runs on localhost — not exposed to internet by default
  • No cloud services required — fully air-gapped capable
  • Data stays in your PostgreSQL instance
  • Auth can be added via Fastify plugins if needed

"What if Claude/Codex add their own memory?"

Short: Great — Khef will integrate. This is complementary, not competing.

  • Built-in memory will likely be per-assistant, not shared
  • Khef works across Claude AND Codex simultaneously
  • You control the data and structure
  • Can export/migrate if something better comes along

"Why should I trust this? It's a side project."

Short: I've been using it daily for weeks. It works.

  • Dogfooding — I use it for all my projects including this one
  • Simple stack — nothing exotic, easy to maintain
  • Self-hosted — you control the data
  • Worst case: export your data and move on
